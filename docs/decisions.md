**Store: Supabase (Postgres).**
Chosen over Neon. Both are Postgres; Supabase adds a built-in data API and access controls that make exposing data to a non-technical dashboard easier, saving plumbing we'd otherwise build ourselves.

**Runtime: daily batch pull via REST, on a cloud cron.**
Not a webhook, not the AmpleMarket MCP. A daily batch satisfies the "keep it up to date daily" requirement. Cloud cron so nothing depends on anyone's machine being on. MCP was rejected because research showed it just wraps the same REST data — no extra capability, worse fit for a scheduled job.

**Message body: HubSpot email bodies captured in v1 via REST (2026-07-15); AmpleMarket message text still v2/webhook. (Updates the original "stays null until v2" decision below.)**
Original stance: bodies are a v2 goal needing an always-on webhook, so the `body` field exists in schema but stays null in v1. **Revised for HubSpot emails specifically:** HubSpot's Search API already returns `hs_email_html` (full) + `hs_body_preview` (snippet) with no extra scope, so we now capture them in the daily pull into `raw_hubspot_emails.body_html` / `body_preview`. Cheap, low-risk, and it (a) gives the strongest same-send dedup signal and (b) banks the raw material for the tailored-vs-generic quality lens. **Going-forward only — no backfill** (older rows stay null; HubSpot retains bodies so a backfill remains possible). Decisions taken with the PM: capture both preview+full, going-forward only, no separate privacy sign-off (treated like other stored activity data). **Still v2:** AmpleMarket LinkedIn/email send text (REST exposes tasks, not send bodies) still needs the Sequence Stage webhook, which only captures from when switched on — no backfill.

**Source of truth split: AmpleMarket for its channels; HubSpot for manual emails + meetings only.**
AmpleMarket activity syncs into HubSpot, so counting both would double-count. AmpleMarket owns LinkedIn, calls, and sequenced emails; HubSpot contributes only the non-AmpleMarket activity (manual Gmail emails, meetings). HubSpot ingestion excludes `lemwarmup` and `amplemarket`-sourced records.
**Superseded (2026-07-13):** AmpleMarket-sourced emails are KEPT (they're the only record of sent sequence emails) — see the later entry.

**Deals excluded — model activities against company + contact only.**
Activity attaches to contacts and companies. Deals are a separate pipeline/revenue object and answer a different question (did activity create pipeline?). Out of scope for v1; possible future attribution layer.

**Coverage source: HubSpot `target account owner`.**
Confirmed by Ray as the field telling us which rep owns each account. One owner per account.

**Account de-duplication by domain is mandatory, upstream of coverage.**
Ray confirmed HubSpot sometimes stores duplicate company records. Left unhandled, one account's activity splits across copies and coverage/depth numbers distort. Collapse duplicates by normalized domain before computing any coverage metric.

**Identity resolution: deterministic-first, fuzzy fallback, cached.**
Person = exact lowercased email; company = normalized domain; fuzzy name matching only for records that fail exact keys; never collapse companies on free-email domains. Cached in a crosswalk so daily runs are incremental joins, not re-resolution.
**Superseded (2026-07-15):** as built there is NO fuzzy matching and NO persisted cache — deterministic full-rebuild each run, with a name-token guard on company merges; see the Phase 2 entry below + spec §2/§9.

**Seniority from job title: rules-first, LLM fallback, cached.**
Dictionary/regex handles common titles; only unmatched/ambiguous titles hit an LLM; every (raw title → bucket) mapping is cached so a title is classified once ever. Raw title preserved, never overwritten.

**Automated-vs-manual flag: AmpleMarket authoritative, HubSpot proxy.**
AmpleMarket exposes an explicit `automatic` flag (high confidence). HubSpot has no clean sequence flag, so its activities get a lower-confidence inferred label.

**Supabase security settings.**
Data API on (needed for the dashboard); auto-expose new tables OFF (data is sensitive; expose deliberately); automatic RLS ON (no table accidentally wide-open). Sensitive internal rep + prospect data justifies the cautious defaults.

**DB schema managed via Git migrations from Phase 3 onward.**
Connect Supabase↔GitHub at Phase 3 (when tables first exist), not before — nothing to sync until then. Keeps every schema change tracked and reversible instead of manual UI edits. Credentials/keys never committed; use secrets/env vars.

**Phase 3 de-duplication & direction rules (confirmed, discovered during Phase 1 rep validation).**
The raw layer deliberately keeps everything; these are applied when building the unified activity model:
1. **Sent vs received emails** — only outbound (sender = rep) counts as rep *effort*; inbound prospect replies are tracked separately as an engagement/outcome signal. (Yianni 2026-07-13: 3 sent, 2 received.)
2. **Same-email cross-tool duplicates** — the same outbound email is logged into HubSpot by multiple tools; treat all copies as one send. **Correction (2026-07-14, Andrew Bell): an exact "subject + date + time" match does NOT work** and would ship near-useless dedup — (a) Apollo rewrites the subject with a `[Apollo] [Email] [<<] ` prefix so subjects differ; (b) the copies land seconds apart (e.g. 10:53:28 vs 10:53:38), so timestamps are never equal; (c) it is N-way across a *varying* set of sources (Apollo+Gmail, Apollo+CRM_UI, up to 3 copies), not a fixed Apollo↔AmpleMarket pair. Correct key: **normalized-subject (strip tool prefixes) + sender + time-window (≈±60s)**, with the message **`body` as the strongest join signal** now that we capture it. (James Falconer 2026-07-13: "Re: VLMs in identity", double-logged, was the first instance.) Ties into the open Apollo decision. **Superseded (2026-07-15):** the production audit disproved this key too — sequence blasts share subject AND body across recipients, subjects mutate beyond tool prefixes, copies span >60s. The corrected key (sender + external-recipient overlap + normalized body + ±180s) is in spec §3; see the audit entry below.
3. **AmpleMarket task↔send reconciliation** — AmpleMarket exposes email tasks but not sends (sends only via the HubSpot copy); reconcile so a task and its send aren't double-counted, without dropping sends.
4. **Call↔task overlap** — a `phone_call` task and its `/calls` record are the same call; collapse via `task_id`.
Deferred here (not in raw) because it needs identity resolution (Phase 2) and the unified model to do correctly.

**Phase 3 rules — additions & one correction (found 2026-07-14 cross-checking Joe Turner, Dillon, Kamil, George Lim). Supersedes rule 4 above.**
- **Attribute emails by SENDER address, not `hubspot_owner_id`.** owner_id is NOT reliable rep identity: it is sometimes shared by several reps at once (owner_id `538916758` carried sent mail from Kamil, Yianni, George Lim, Nico, Ilaria) and one rep's mail can span many owner_ids (Dillon across 6). Determine who sent an email from `hs_email_from_email`, matched to the rep's linked addresses (the `/users` map). Direction (sent/received) then follows: sender ∈ rep's addresses ⇒ outbound effort; else inbound reply. Some reps happen to have a clean owner_id (Yuvi, Katie, Joe) but you must not depend on it.
- **Calls — count attempts vs conversations separately; `task_id` is usually null so DO NOT collapse on it (this corrects rule 4).** On `/calls`, `task_id` was null for all of Joe's calls, so the "collapse via task_id" method fails in practice. A rep pursuing one contact logs several call records seconds apart, often to *different phone numbers* (Joe 13 Jul: 4 calls in 123s to 4 different numbers for the same contact Iyad Ahmad = 4 genuine dial attempts, 1 real conversation — rep confirmed). Group a "conversation" by **same contact + tight time window**, not `task_id`. A record is a real human conversation only when `human = true`; `answered = true` alone can be voicemail/IVR. Report both **dial attempts** and **connects/conversations** — collapsing to one hides effort; treating every record as a conversation inflates volume. (The original rule 4 — that a `phone_call` task and its `/calls` record are one call — still holds *when* a task_id is present; it just cannot be the primary key for de-dup.)
- **Calendar-invite fan-out — collapse.** One calendar action is written once per attendee (Dillon 13 Jul: 6 identical Apollo "Updated invitation" rows, same timestamp, one per invitee). Collapse to a single activity keyed on subject + time (differs only by recipient), or it 6×-inflates. **Reshaped (2026-07-15):** per-attendee fan-out proved rare (this was the only group in 5 days; most invite rows pack all attendees into one `to_email`) — invites dedupe via the corrected key in spec §3; subject+time collapse survives only for invite-classified rows.
- **Exclude internal-only recipients.** Activity addressed solely to internal `@encord.com` people is not outbound CA activity (Dillon's invite attendees were all internal) — don't count it as outreach.

**Phase 3 edge cases — further additions (found 2026-07-15 doing itemized body-level pulls: Andrew Bell, Yuvi, Kat Romanska).**
- **A fourth email source exists: `CRM_UI`.** Beyond `Amplemarket` / `Apollo Integration` / manual Gmail (`EMAIL`), HubSpot logs emails sent from inside its own web UI as `hs_object_source = CRM_UI` (seen on Andrew Bell's "Re: Encord x TUM"). It is a manual send. The source enumeration and the automated-vs-manual inference must include it; any unrecognized source value defaults to manual and is never dropped.
- **Calendar invites are the same event as their meeting object — don't double-count across channels.** A tool-booked meeting writes **both** an "Invitation:"/"Updated invitation:" row in the emails table **and** a row in the meetings table (Andrew Bell 2026-07-14: "Encord x Bosch | Sync" invite email at 12:00 BST = Bosch meeting object at 11:00 UTC). Invite/notification emails belong to the meeting channel, not counted as separate outbound emails, or every tool-booked meeting inflates both the email and meeting tallies. (Distinct from invite fan-out, which is duplication *within* the email table.)
- **One rep can appear as multiple AmpleMarket accounts, in two shapes** — one account with multiple mailboxes (Katie: one id, `@encord.com`+`@encord.ai`) *or* multiple separate accounts one mailbox each (Yuvi: two distinct `/users` ids). Union **all** of a person's `user_id`s before aggregating their calls/tasks, or one rep splits into two.
- **A call's `contact` can be null even for a real conversation** (Andrew Bell's one `human=true` connect had no contact) → coverage logic must keep unattributable-but-real calls, not drop them.

**Rep identity — the technical ID→person map comes from AmpleMarket `/users`; only "who is a CA" needs Ray. (Refines the entry above; found 2026-07-14.)**
Two separate questions were being conflated into one "roster" blocker: (1) *which AmpleMarket internal ID belongs to which person, and which email addresses are theirs* — a technical mapping; and (2) *which of those people count as CAs* — a team-membership call. Finding: `/users` answers (1) directly and completely — it returns each rep's `id`, `email`, and a `mailboxes` array that links their multiple addresses (e.g. Yuvi's `@encord.com` + `@encord.ai`), and it lists reps **regardless of whether they logged any tasks.** So the technical map is buildable today; only (2) needs Ray / sales leadership. **Why this is critical:** AmpleMarket `/calls` records identify their rep by internal ID only — no name or email — so calls cannot be attributed to a person by any means *except* this map. The earlier stopgap (inferring a rep's ID from their completed tasks) silently dropped every call for any rep with no tasks that day. Evidence: Yuvi had 0 tasks on 13 Jul, so his 2 answered calls (Mediaire, Siemens Healthineers) were invisible until mapped via `/users`, then matched exactly what he reported. Note also: a rep can have >1 `/users` account (Yuvi has a dormant `@encord.ai` account plus his active `@encord.com` one) — link them, but attribute activity to the account that actually carries it.

**Phase 1 raw ingestion: land faithful per-source copies, keyed by source id, idempotent.**
Four raw landing tables (`raw_amplemarket_tasks`, `raw_amplemarket_calls`, `raw_hubspot_emails`, `raw_hubspot_meetings`) plus an `ingestion_runs` audit log. Each row stores a few extracted columns for convenience plus the full source payload in a `raw` jsonb column, so nothing is lost before Phase 3 normalization. Primary key = source id; `INSERT ... ON CONFLICT DO NOTHING`, so re-running a day inserts zero duplicates. AmpleMarket ignores date filters, so we page newest-first and stop once we cross below the target day (tasks keyed on `finished_on` with `status=completed`; calls on `start_date`).

**HubSpot email exclusions at ingestion — warmup noise only. (SUPERSEDES the original "exclude AmpleMarket-sourced" rule below.)**
Keep every HubSpot email regardless of origin (each tagged via `object_source` / `object_source_detail_1` = Amplemarket / Apollo Integration / manual Gmail). Skip only warmup noise (subject contains `lemwarmup`/`lemwarm`/`amplemarketwarmup`/`warmupemail`). Exclusion counts recorded per run in `ingestion_runs`.

**Phase 1 ingestion script — 3 follow-up fixes found in code review (2026-07-14, queued as TODO in roadmap Phase 1, not yet applied).**
1. **Lookback window too short — late-arriving records are silently lost forever.** The script only ever pulls "yesterday," but some HubSpot/AmpleMarket records land hours after that snapshot is taken. Evidence: querying HubSpot live on 14 Jul for 13 Jul returned 1,379 emails vs the 1,377 we captured; the `ingestion_runs` log shows 13 Jul AmpleMarket tasks grew from 185 → 304 fetched across successive re-runs of the *same* day. Fix: widen each daily run to re-check the last ~3–4 days, not just yesterday — safe because re-runs are proven idempotent (0 duplicate inserts observed across every repeat run in the log).
2. **We never capture who an email was sent to, or meeting attendees/outcome.** Inspecting the actual stored `raw` jsonb confirms it only contains the specific properties requested (`hs_email_from_email`, subject, timestamp, source, owner) — not a recipient field for emails, nor an attendee/outcome field for meetings. Concretely: a "booked meeting" today only means one was scheduled — we don't know if it was actually held, cancelled, or a no-show. This blocks Phase 3 rules already committed in this doc and the roadmap: internal-recipient exclusion and invite-fan-out collapsing both need the recipient list; attaching an outbound email to the prospect it went to needs it too; and meeting→rep attribution has no path at all without attendees, since `owner_id` is separately proven unreliable (see rep-identity entries above). Fix: add the missing properties to `ingest_hs_emails`/`ingest_hs_meetings`'s requested `props`, then re-run ingestion once for the days already captured (11–13 Jul) to backfill — cheap, since HubSpot still has the data; every day this isn't fixed is a day added to that backfill.
3. **One job's failure aborts the whole day; plus a dormant-repo scheduling risk.** `main()`'s job loop re-raises on the first exception, so e.g. a HubSpot outage stops the AmpleMarket jobs from running too, and that day never gets a second attempt (partly mitigated by fix #1's lookback). Fix: let each of the 4 jobs (AmpleMarket tasks/calls, HubSpot emails/meetings) fail independently and keep going. Separately (not a code fix): GitHub Actions auto-disables scheduled workflows after ~60 days with no repository activity — worth a periodic manual check if the project goes quiet.
Lower-priority, noted but not queued: the AmpleMarket pager's 200-page safety cap would silently truncate a large backfill rather than erroring; retry logic is only 4 attempts with short backoff; ~~meetings are insert-only~~ (fixed with the batch below — meetings now update on conflict).

**All 3 code-review fixes applied 2026-07-15 (supersedes the TODO status above). Structural facts learned while applying them:**
1. **Lookback:** scheduled runs (no date argument) now sweep the last 3 days (`LOOKBACK_DAYS = 3`) (**Superseded 2026-07-15:** now 5 — see the production-audit entry); an explicit date argument still runs that single day (manual/backfill behavior unchanged). Proof of value on the very first run: it recovered 72 late-arriving 13-Jul tasks + 3 late 13-Jul emails and 3 of 12-Jul, all previously lost; zero duplicate inserts.
2. **Recipients/attendees/outcome:**
   - Emails now request + store `hs_email_to_email` → `to_email`, `hs_email_cc_email` → `cc_email` (bcc requested into `raw` only). **Format facts:** multiple recipients are **semicolon-separated** in one string; individual entries vary between `a@b.com`, `Name <a@b.com>`, and `<a@b.com>` — Phase 3 must parse/normalize before matching. A rep's own address often appears among the recipients (self-cc pattern).
   - Meetings now request + store `hs_meeting_outcome` → `outcome`, `hs_attendee_owner_ids` → `attendee_owner_ids` (semicolon-separated **owner ids of internal attendees** — this is the usable meeting→rep attribution path, replacing the unreliable single `owner_id`), plus `start_time`/`end_time`. **Honest limitations:** `outcome` is only populated when a rep/tool sets it (22 of 105 stored meetings have one: CANCELED/SCHEDULED/COMPLETED) — null outcome means *unknown*, not "didn't happen"; **external (prospect) attendees are HubSpot associations, not properties — NOT retrievable via the Search pull** — prospect-side meeting participation still needs an associations call if ever required.
   - **Meetings switched from insert-only to update-on-conflict** (title/outcome/attendees/times/raw refresh on re-runs) because outcome mutates after first capture; combined with the 3-day lookback, an outcome set the day after a meeting is now picked up automatically. The other three raw tables stay append-only (`DO NOTHING`) — their records don't mutate.
   - One-time backfill (UPDATE by id, no inserts/deletes) filled `to_email`/`cc_email` — and bodies where null — on all previously-ingested days: 11 Jul (78), 12 Jul (33), 13 Jul (1,328), 14 Jul (1,294), 15 Jul (337 rows).
3. **Per-job error isolation:** each of the 4 jobs (× each lookback day) now fails independently — the failure is rolled back, logged to `ingestion_runs` (`status='error'`), and the remaining jobs still run; the process exits non-zero at the end so GitHub Actions still flags the run. Note the rollback matters: a failed Postgres write leaves the connection in an aborted state that would poison subsequent jobs without it.

**Phase 1.5 — HubSpot Company/Contact mirror: dimension semantics; companies live now, contacts gated on a Supabase upgrade (2026-07-15).**
Companies + contacts are **entity/dimension data**, not events, so they're built differently from the four activity tables: update-on-conflict (keep current state — tiers, owners, titles change), no `activity_date`, no daily date sweep; synced **incrementally by last-modified watermark** (first run = full mirror, then deltas; the 10k-per-Search-query cap is beaten by sorting ascending on last-modified and restarting from the watermark). Entity sync runs only on scheduled (no-date-argument) runs — explicit-date runs are activity backfills and skip it.
**Sizing decision (superseded same day — see below):** measured live: 154,369 companies (~100 MB payload) + 446,354 contacts (~300 MB) against a Supabase **free tier (500 MB cap)** with the DB at 40 MB. Blowing the cap would block the live daily pipeline. First answer was a `CONTACTS_SYNC_ENABLED` gate awaiting a plan upgrade; replaced the same day by the activity-scoped design below, which needs no upgrade.

**Contacts: ACTIVITY-SCOPED mirror, not a full mirror (PM decision 2026-07-15; supersedes the gate above).**
Companies stay a FULL mirror (154k — filtering on `target_account_owner` would silently drop accounts whose CRM assignment is missing/stale, a failure mode we've already proven with `hubspot_owner_id`). Contacts are mirrored **only if they appear in real rep activity**: distinct emails from AmpleMarket task/call contacts + HubSpot email senders/recipients (parsed from the semicolon/`Name <a@b>` formats), minus internal rep domains, fetched via HubSpot **`batch/read` with `idProperty: email`** (100 per call; two aliases can resolve to one contact — dedupe by id before upsert), fully re-read each run so jobtitle/company stay fresh. **Self-extending:** touching a new person pulls them in on the next run. **Accepted trade-off:** never-touched contacts are absent → "untouched contacts per account" (Phase 5 whitespace denominator) needs a Supabase upgrade + full mirror later; nothing else is lost. First live run: 2,897 activity emails → 2,869 matched (99%), 2,850 joinable to a held company; DB at 171 MB of 500.

**Rep sending domains include `tryencord.com` (found 2026-07-15 investigating Ray's direction question).**
Ray flagged AmpleMarket emails with direction looking wrong in HubSpot. Root cause found: three reps send cold outreach from a **dedicated outreach domain `@tryencord.com`** (annabel, george, tom.kennedy — 327 emails in-store), which our rep-address logic didn't know, so their outbound looked like inbound from strangers. The rep-identity address set (Phase 2) must include `encord.com`, `encord.ai`, **`tryencord.com`** (now the `INTERNAL_DOMAINS` constant in `ingest.py`, used to exclude reps from the contact mirror) — and these outreach addresses likely do NOT appear in AmpleMarket `/users` mailboxes, so the `/users` map alone is not sufficient for rep identity. Watch for more sending domains appearing.
**Key facts learned:** `target_account_owner` is an **owner id** (same id-space as `hubspot_owner_id`), present on only **~1,515 companies** — the coverage target universe. Contacts' last-modified property is `lastmodifieddate` (companies use `hs_lastmodifieddate`). Contact→company primary association is the plain property `associatedcompanyid`. Several ICP company fields are deprecated duplicates — the current ones are `account_icp_tier_validated`, `account_icp__tier_new`, `vertical__aligned_by_team`.

**AmpleMarket's task/call feed is eventually consistent — the lookback width is evidence-driven, not guaranteed. (Found in code review 2026-07-15; hardening applied same day.)**
Proof from `ingestion_runs` + live re-pulls of one fixed day (13 Jul tasks): a run at 12:00 found 185; a re-run at 13:00 the same day found +119 more; another sweep a day later found +72 more — all with `finished_on` genuinely inside 13 Jul, spread across the whole day. So the API keeps surfacing a *finished* day's records for **2+ days afterwards**, and its newest-first ordering is not stable. Two consequences, both handled:
1. **Paging stop-rule hardened:** the ingester previously stopped at the *first* record older than the target day; with unstable ordering that can miss records hiding behind the stop point. It now stops only when a **whole page** falls below the target day. Same for calls.
2. **`LOOKBACK_DAYS` is a monitored bet, not a proof** (3 then; widened to 5 by the 2026-07-15 audit — arrivals measured at day+3). The check: if the **oldest** day of the sweep still reports `rows_new > 0` regularly in `ingestion_runs`, records are arriving later than the window reaches — widen it. This check should become an automated alert in a later phase; until then it's a manual glance at the run log.
Also from the same review: network-level errors (connection reset / DNS / timeout) are now retried like 5xx errors (previously only HTTP 429/5xx were, so a blip failed the job on the first try), and the GitHub Actions workflow gained a concurrency guard (a manual run can no longer interleave with the 6am cron) plus a 30-minute timeout.

**Phase 3 built: unified activity model (`model/`) + Git-managed migrations (`migrations/`) + rule tests on real fixtures (`tests/`) — 2026-07-16.**
One `activity` fact table (one row = one real activity event, spec §2 grain), rebuilt deterministically from raw + identity tables by `model/build_activity.py`; all dedup/attribution judgment lives in `model/rules.py` as pure, DB-free functions so tests can replay real cases through the exact production logic. Structural decisions:
- **Nothing is dropped — the audit invariant.** Every raw activity row lands in EXACTLY ONE activity row's `source_ids` (verified at the end of every build; a violation aborts before anything is written). Rows that must not be counted stay present with `counts=false` + a named `excluded_reason` — so any dashboard number is auditable down to raw records (the spec §6 transparency requirement), and "excluded" is always visible, never silent.
- **Shadows, not joins, for the two cross-system overlaps (spec §3 rules 7/8):** AmpleMarket *email* tasks (the send's countable record is the HubSpot-synced copy) and *phone_call* tasks (the countable dials are `/calls` records) are kept as non-counted `email_task`/`call_task` rows. Row-level task↔send joins were rejected — same reasoning as invite↔meeting: unreliable keys, and the shadow keeps the evidence visible if a sync ever drops a send.
- **Dimension views, not copied tables:** `dim_account` / `dim_contact` / `activity_flat` are zero-copy VIEWS over the raw mirrors + crosswalks (migrations/001) — no second 154k-row company copy to keep fresh; the fact table is the only new stored data.
- **Inbound engagement counts only when a CA is among the recipients** (to ∪ cc ∪ bcc), attributed to that CA; inbound to AEs/others is kept with `excluded_reason=inbound_no_ca_recipient`. Inbound dedup overlap is computed over ALL recipients (a prospect reply's recipients are our own reps — the outbound external-only rule would find nothing to overlap).
- **Meetings count when a CA is among the internal attendees** (`attendee_owner_ids` ∪ owner, resolved against the roster; ALL attending CAs stored in `ca_ids`, first = `ca_id`). Known accepted limitation: external attendees are unavailable (Phase 1 finding), so internal-only meetings are counted too — a future associations pull can fix this; outcome (incl. CANCELED) is exposed on the row so Phase 4 can filter.
- **Body-key robustness — two live refinements to the corrected dedup key (2026-07-16, found on first live build):** (a) mail-client banners injected into one tool's copy but not the other's ("This is the first time you're receiving an email from…" — Gmail) are stripped before the body prefix is taken, or true duplicate inbound replies never merge (live: the Lucid Bots reply pair); (b) one copy's preview sometimes carries a trailing signature block the other's stops before — so two normalized bodies also match when one is a prefix of the other (min 20 chars; live: Satchel's "try now." vs "try now. -- Kind regards…"). **"Meetings booked" is NOT "meetings held" — booked/canceled/held must be split before it's used as a conversion denominator (found 2026-07-16, PM analysis of Andrew Bell).**
Investigating why Andrew Bell showed 13 meetings in 5 days (highest on the team) surfaced that the raw meeting count is **bookings, not occurrences**, and is inflated by three things: (1) **cancellations** — 4 of his 13 carry `outcome=CANCELED`; (2) **rebookings of the same prospect** — e.g. "Manish" booked twice in one day, both canceled, counted as 2; and (3) **possibly-internal events** slipping in (a generic "Encord Workshop" with no external attendee). Cross-checking his own follow-up email bodies ("great to meet earlier" = held; the Lucid Bots prospect wrote "we're not in a place to start talking labelling right now" the hour before = almost certainly not held) put the realistic **held** count at ~4–7, not 13. Root cause is a known hygiene gap already noted in the audit: reps log `outcome` on only ~11 of 55 meetings (~20%), so most meetings are `outcome=null` = *unknown*, not *held*. **Implications for Phase 4+ (the aggregate/coverage views):**
- Never present a single "meetings" number. Surface **booked vs canceled vs held (vs no-show vs unknown)** as separate figures — the `outcome` column already carries COMPLETED/CANCELED/SCHEDULED/RESCHEDULED/no-show; null must render as "unknown," never folded into "held."
- Any **meetings-as-conversion / meetings-per-activity** metric must use *held* (or at minimum booked-minus-canceled) as the denominator, or it flatters booking-link-heavy reps whose slots cancel and rebook often (Andrew's are mostly "X and Andrew Bell | Encord" booking-link meetings, which churn by nature).
- The dedup already collapses duplicate meeting OBJECTS (spec §3 rule 10), but that's distinct from this: two *different* bookings for the same prospect are genuinely two rows and should not be merged — they should be visible as booking churn.
- **This does not overturn the Andrew finding** — even 4–7 held meetings off ~27 human actions still converts far better than volume reps (Joe: 3 meetings off ~560 activities), and the SAO tracker independently corroborates (25 SAOs, ~92% self-generated). It sharpens it: the *denominator* was soft, the *direction* holds.
Cheapest fix when Phase 4 lands: derive a `meeting_status` from `outcome` (held / canceled / scheduled / unknown) and default every meeting rollup to held-only, with booked/canceled shown alongside. **Also worth raising with sales leadership as a process item** — outcome-logging discipline (~20%) is the single biggest quality gap in the meeting metric, and no amount of pipeline work can recover an outcome a rep never logged.

**Internal meetings barely pollute the count — but by rep behaviour, not by a pipeline rule (verified 2026-07-17).**
Nothing in the model checks that a counted meeting has an external (non-Encord) attendee — it can't, because the meetings API pull doesn't include guest emails. A manual read of all 126 CA-counted meeting titles across Jul 6–15 found only ~3–4 plausibly internal events (~3%: a "Global Sales Enablement Bi-Weekly", a leaving-drinks, one ambiguous workshop); everything else follows the client-facing naming pattern ("X <> Encord | demo/intro") or came via RevenueHero (prospect-booked by definition). Conclusion: CAs simply don't log internal meetings in HubSpot, so the number is clean **today** — but this is protected by habit, not by code, and could silently degrade if logging behaviour changes. Watchlist item, not a fix-now. The proper guard (require ≥1 external attendee) arrives for free with the Phase 6 attendee-data work, which the booked-vs-held fix needs anyway.

## 2026-07-19 — Phase 4 (rep scorecard) design decisions

- **Shape: one parameterised SQL function, not stored tables.** `rep_scorecard(start, end)` (migrations/002) computes per-rep totals + coverage on demand from `activity_flat`; preset views `rep_scorecard_7d/_30d/_alltime` wrap it with rolling windows (`current_date` evaluates at query time). Nothing stored, nothing to keep in sync, no nightly step added — and the future dashboard calls the same function with any custom range, so PM reports and dashboard can never disagree.
- **Strictly additive and separable (PM requirement):** the migration only CREATEs new objects; dropping the function + three views restores the pre-Phase-4 state exactly. No Phase 0–3 file was modified.
- **Meetings: primary-CA attribution kept** (`ca_id`, first of `ca_ids`) — same attribution every validated audit workbook used; crediting all attendees would double-count (4 of 162 meetings are multi-CA; one has 7 CAs). Buckets held (COMPLETED) / canceled (CANCELED) / scheduled (SCHEDULED, RESCHEDULED) / **unknown = booked − the rest**, so a novel HubSpot outcome value lands in *unknown*, never in *held*, and the four buckets always sum to booked by construction.
- **Coverage semantics:** owned = `dim_account.owner_rep_id` (target-account owner; owner sets are disjoint across reps, so team-level sums are valid); touched-owned counts only accounts touched **by their owner** — a colleague touching your account is not your coverage. Verified live 2026-07-19: 1,612 owned accounts, 1,602 by roster CAs; Constantin 81/135 = 60% hand-checked.
- **Pursuit counts must come from the function, not day slices:** pursuits = distinct `call_group_id` in the window; a dial burst crossing midnight would double under per-day summation.
- **Validation:** `rep_scorecard('2026-07-06','2026-07-09')` reproduces the Jul 6–9 audit workbook **exactly for all 17 reps** (auto/manual/dials/pursuits/conversations/LinkedIn/meetings/inbound); Excel render (`scorecard/build_scorecard.py`) column-sums reconciled against the DB before delivery.
- **Ops gotcha (self-inflicted, worth remembering):** psycopg2 `set_session(readonly=True)` on the Supabase **transaction pooler** leaks the read-only flag to other clients sharing the pooled server connection — it made `migrations/apply.py` fail with `ReadOnlySqlTransaction`. Rule: never set session-level state through the pooler; read-only scripts just run SELECTs.
